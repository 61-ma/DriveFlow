## PM 思考方法论（贯穿全文档）

每个产品决策都会回到下面 6 个 PM 灵魂问题：

| 问题 | 出处 | 在本项目的应用举例 |
|------|------|------------------|
| **Why now？为什么现在做？** | Lenny PM Skill: writing-prds | Robotaxi 商业化窗口期，谁先定义车内 AI 体验谁拿用户心智 |
| **Struggling Moment：用户挣扎的具体时刻是什么？** | Lenny PM Skill: problem-definition (Bob Moesta) | 急刹车后那 3 秒的恐慌、深夜独自上车后的孤独感、60 岁阿姨第一次面对无司机的不知所措 |
| **What's the North Star？北极星指标是什么？** | writing-north-star-metrics | 主指标：行程满意度；反指标：主动服务被拒绝率（防止单维度优化）|
| **Who wins, who loses？谁是赢家谁是输家？** | evaluating-trade-offs (Ramesh Johari) | 选猫形象：年轻女性赢，但中年男性可能觉得幼态——用"商务静默模式"对冲 |
| **Reversible or one-way door？决策是否可逆？** | evaluating-trade-offs (Annie Duke) | 形象 IP 是 one-way door（一旦上线难改）；技术栈是 reversible（可重写）|
| **Hidden cost：隐性成本是什么？** | evaluating-trade-offs (Stewart Butterfield) | 引入 PixiJS 可见成本是 200KB，隐性成本是「不再是单文件」「面试官打开复杂度」|

```

**【我的判断】**（PM 视角，否决 AI 建议）

- AI 视角太"产品能力"，PM 视角应该是"市场窗口 + 用户心智"。下一轮要把 AI 拽回到"无人车的特殊性"。

### 第 3 轮：纠偏 ——「Struggling Moment」

**【我】**
```
你的分析有个关键盲区：

Robotaxi 和普通网约车最大的不同是——没有司机。我用 PM 的"Struggling Moment"框架重新分析：

**用户的挣扎时刻不是"我想更高效"，而是：**
1. 上车后 30 秒——突然意识到"没有人在前排"，孤独 + 不安
2. 急刹车那 0.5 秒——心跳加速，"刚才发生什么了？"
3. 深夜目的地是住宅区——"下车后这条路安全吗"
4. 60 岁阿姨第一次坐——"我要怎么操作？谁能帮我"

这些是真实的 struggling moment。AI 助手如果连这些都没回答，就是"不存在"。

所以我的产品假设是：**无人车乘客的核心需求不是效率，是安心 + 陪伴。**

帮我想反面论据，别只附和——什么样的用户/场景下这个假设会**完全错**？
```

**【AI】** 这次回复质量高了很多（被"想反面论据"激发了批判性）：
> 反面论据：
> 1. **商务人士**只想安静（需要"静默模式"）
> 2. **短途用户**（5 分钟）不需要陪伴（需要行程时长触发阈值）
> 3. **"情感伴侣"对部分用户像幼稚化**（需要可关闭 + 形象选择）
> 4. **恐怖谷风险**：陪伴感做不到位，反而比"没有"更糟

**【我的判断】**（PM 视角）
- 这 4 条反面论据**就是产品设计要对冲的风险**。逐条转成产品规则：
- 第 1 条 → **商务静默模式**（详见 02 文档），默认关闭主动打扰
- 第 2 条 → 主动服务**只在行程 >15 分钟**时触发
- 第 3 条 → 不做"恋爱型"，做"治愈宠物型"（猫的接受度比人形 AI 高）
- 第 4 条 → 这正是我选猫的理由（恐怖谷风险远低于虚拟人）
- **Who wins, who loses**：年轻女性赢、年轻男性中性、中年商务可能输——商务静默模式是补偿设计。

**【我】**

```

**【AI】** 推荐了虚拟人：

**【我的判断】**（PM 视角，否决 AI 建议）
- AI 推荐基于"技术能力维度"。但 PM 视角应该看**4 个维度的综合得分**：

| 维度 | A 人形虚拟人 | B 动物（猫）| C 抽象光球 |
|------|------------|-----------|----------|
| 第一眼治愈感 | 中（"恐怖谷"风险）| **高**（天然治愈）| 低（无情感锚点）|

- **Reversible 评估**：形象 IP 是 one-way door，但"猫的具体长相"还可以微调。先定方向（猫），细节留后。

# 第三部分：北极星指标与上线策略（PM 视角扩展）

### 第 22 轮：动画原理讨论（理解原理再决定做多少）

**【我】**
```

```

**【AI】**
> **Live2D 本质：**
> 1. 将 2D 图拆成多个 mesh 区域
> 2. 对每个区域做独立微小变形（头部偏移、身体呼吸膨胀、耳朵摇动）

**【AI】** 用 OpenCV inpainting：① 找到橙色主体连通块作为"桔子身体"mask；② 在 mask 之外检测白色 / 浅灰像素 → inpaint 为透明；③ 在桔子下半部分（y > 桔子高度 65%）的白色像素 → 用周围的橙色 inpainting，把 logo 和文字"擦除"。

**【我的判断】**（PM 视角）
- 这不是"美工修图"问题，是**IP 资产中立化**问题——桔子 IP 上**带具体 logo / 文字**会让它变成"滴滴桔子"而不是"小桔"，限制了未来跨品牌复用空间
- 把 logo / 文字去掉后，**品牌色（橙）成为唯一的品牌识别物**，符合"形象先于功能"的设计原则
- AI 用 OpenCV inpaint 而不是 Photoshop 让我能在 git 里追踪修改，并保留 `mascot-orange.original.png` 备份做可逆决策；交付前再清理掉备份文件，把交付包体积从 1MB 降到 502KB

### 第 33 轮：场景模拟面板 ——「让评审看得见全貌」

**【我】**
```
demo 里只有 3 个情绪按钮和麦克风按钮，评审看不出 PRD 6 大功能在哪。
把 PRD 上要展示的功能做个面板，可以用点击模拟输入。
```

**【AI】** 设计 Tab 式场景模拟面板：6 个 Tab（智能客服 / 情绪陪伴 / 主动服务 / 城市探索 / 应急 / 形象状态）× 32 个场景按钮。引入 4 种执行 kind：voice（听 → 思考 → 回）、invite（弹邀请 + 三选项）、emergency（≤0.5s 应急话术）、state（直接切形态）。所有话术从话术库 v2.0 复用。

**【我的判断】**（PM 视角）
- **关键决策**：按钮的 label 直接 = 模拟用户语音输入（"🌡 太冷了"、"⏱ 还有多久到"）—— 让评审能"用用户语言"而不是"产品 jargon"操作
- **拒绝**：不要做成"步骤操作面板"——demo 是给评审看产品力的，不是教评审怎么用车机
- 32 个场景按钮 = PRD 验收标准 14 个 + 话术库 11 个邀请场景 + 6 个形态切换的合集，**确保 PRD 每条都"演得出来"**

### 第 34 轮：面板布局重构 ——「演示控制台 vs 产品本身的边界」

**【我】**
```
场景模拟可以做成下方长条。
```

**【AI】** 把 demo-card 从左侧 panel 抽出，改造为屏幕底部独立的 `.demo-bar`：left: 32px / right: 32px / bottom: 28px / height: 172px。左侧 300px 放 6 个 Tab + 描述，右侧自适应宽度铺 4 列按钮。`.left-panel` 和 `.right-stage` 的 bottom 改为 218px 给底部条让位。

**【我的判断】**（PM 视角）

- 当评审操作 demo 时，"看产品 UI"和"操作演示控制台"是两件事——把它们空间分离 = 让评审注意力分配更清晰

### 第 35 轮：形态系统扩展 ——「PRD 写了不等于做了」

**【我】**
```
小桔 IP 可以优化下多种形态。
```

**【AI】** PRD 3.1 里写过 6 种情绪状态 + 4 种"场景化形变"（雨天 / 夜晚 / 拥堵 / 守护），但 V2 demo 只实现了 6 种基础情绪。AI 提议补齐：
- 6 情绪：idle / happy / curious / listening / sleepy / alert
- 4 场景：rainy（☂️雨伞 + 蓝色调 + 雨滴动画）/ night（🌙月亮 + 暖橙色）/ explore（🚩旗帜 + 跳跃感）/ guardian（🛡守护盾 + 红色光环）

每种形态加 `decor` 字段，引入 5 类装饰动画（漂浮 emoji / 头顶帽 / 侧面装饰 / 旗子 / 守护盾 / 雨滴）。

**【我的判断】**（PM 视角 + 反思）
- **这是 PRD 完成度的盲区**：PRD 文字里写"雨天有水滴顶帽 + 蓝色调"，我以为"写了 = 做了"，实机调优才发现"写了 ≠ 演得出"
- **PM 教训**：PRD 完成不等于产品可演示——这之间还有"被评审实际操作 1 小时"这条最严苛的检查
- **采纳**：用装饰层系统（emoji + CSS 动画）而不是替换 PNG，因为：① 不丢用户喜欢的桔子形象；② 装饰组合可扩展（未来加新场景只需加 decor 类型）

### 第 36 轮：每次对话都要有回复 + 形态自动切换 ——「不留对话死角 + 让小桔活起来」

**【我】**
```
形态需要自动切换。用户的每一次对话都要有回复。
```

**【AI】** 两个改动绑定一起做：

**A. 每次对话都有回复**：所有 4 种 kind 都保证至少有一条 didi 文字回应。state kind 切换 300ms 后自动 addMessage('didi', randomLine(state))；invite kind 把邀请文案同步写到对话区。每个形态加 `lines: ['…', '…', '…']` 数组（共 24 条独白话术）。

**B. 形态自动切换三层机制**：
- ① 待机微切换：每 7s，idle 状态 + 最近无操作时 45% 概率切到 curious / happy / listening 之一，8s 后自动回 idle
- ② 环境感知 mock：启动后 28s/70s/115s/160s 分别自动模拟雨天 / 夜晚 / 探索 / 沉默，匹配 PRD 3.1 第 5 条的"场景化形变"
- ③ 主动服务弹窗：保留原有 45s 第一次 + 60s × 30% 频率克制策略

**【我的判断】**（PM 视角）
- **"对话死角"原则**：用户感知到任何状态变化（视觉、听觉、点击）都必须有对话区文字回应作为"系统在线"证据——这是车载 AI 比手机 AI 更苛刻的要求（用户没有第二屏验证 AI 是否死了）
- **"活起来"原则**：默认状态下小桔静止 idle 等于死机感。用 7s 微切换 + 时间轴环境模拟两层机制，让评审在 1 分钟内能看到所有 4 种场景化形态自动出现
- **关键约束**：所有自动切换都必须避让"最近 6-8s 内的人为操作"——AI 主动 ≠ AI 抢话

### 第 37 轮：气泡场景化 ——「气泡不是装饰，是接口」

**【我】**
```
桔子的气泡要跟场景匹配起来。
```

**【AI】** 气泡过去固定橙色 + 固定 🍊 图标，所有场景看起来一样。改造方案：
- CSS：`.invite` 用 `--invite-tone` CSS 变量；边框 / 阴影 / 选项按钮颜色都从变量派生；图标用 `attr(data-icon)`
- JS：`STATE_CONFIGS` 每个形态加 `tone` + `icon` 字段；showInvite 根据传入的 state 自动设置 `--invite-tone` 和 `data-icon`
- 优先级：场景显式 icon > 形态默认 icon > 默认 🍊

效果：雨天气泡 = 冷蓝边框 + ☂️ 图标 + 雨天文案；夜晚 = 暖橙 + 🌙；故宫 = 金黄 + 🏯。过渡 400ms 平滑。

**【我的判断】**（PM 视角）
- **重新理解组件**：之前我把"邀请气泡"当**装饰组件**（漂亮就行），调优时把它重新理解为**信息接口**（颜色编码场景、图标编码意图、位置编码紧急度）
- **设计原则切换**：组件设计原则从"美观第一"变成"语义第一"——任何视觉属性都应该承载产品语义

### 第 38 轮：避让交互 ——「让小桔礼貌让位」

**【我】**
```
（截图：气泡盖在桔子的雨伞和蒂叶上）
现在弹出框和图案有点重合，气泡还是会跟图案重合，
可以改下气泡位置或者桔子的位置。
```

**【AI】** 720p 车机屏垂直空间紧张：topbar 56 + demo-bar 218 = 274，剩 446px 给桔子(380) + 气泡(120) — 总和 500，必然重叠。方案：
- 不缩桔子也不挪气泡 —— 让小桔在气泡显示时**自动让位**
- `.mascot-stage.invite-shown { transform: translateY(54px) scale(0.86); }`，520ms 缓动
- showInvite 时 `stage.classList.add('invite-shown')`，hideInvite 时 remove

**【我的判断】**（PM 视角 + 意外收获）
- 这本来是个"修 bug"任务，但**让小桔下移 + 微缩**反而带来了惊喜——视觉上像"小桔在让用户先决策"，强化了"伴侣"的角色感
- **PM 设计教训**：被动修 bug 时也要找"主动设计的机会"——这次的避让动画可能比"完美布局不冲突"更打动评审

### 第 39 轮：代办话术补全 ——「三选项中最被忽视的那一个」

**【我】**
```
气泡内容要跟用户点击相匹配。
```

**【AI】** 排查发现：邀请的三选项中，「好的」有专属 accept 话术，「安静一会儿」是零失落静默，唯独「你帮我选」在所有场景下都返回固定的"没问题，我来挑～"——**11 个邀请共用一句**。

修复方案：每个 invite item 加 `auto` 字段，runScenario 透传，showInvite/hideInvite 用 `inviteAuto`。为 11 个邀请 + 3 条自动主动服务话术分别写专属代办回复。

代办话术设计三原则：① 要具体到执行细节（"挑路口照明最亮的下车点"）；② 要透露代办依据（"看您最近常听爵士"）；③ 要给可量化承诺（"30 元以内"）。

**【我的判断】**（PM 视角 + 设计差异化）
- **车载 AI vs 手机 AI 的核心差异**：手机 AI 给选项让你选，车载 AI 替你选好交付——因为车上你的双手并不空闲（看路 / 看孩子 / 看风景）
- **"信任授权"的具体性**：用户敢选"你帮我选"是因为知道被授权方会做什么—— `auto` 话术不是装饰，是**代办协议**
- 这一轮直接催生了**话术库 v3.0 第 7 章「三选项代办话术」**——把 demo 修复结果反哺到设计文档

**核心信号**：

不是每次 AI 给的东西我都改——
- **有的轮次我全盘采纳**（第 11 / 12 / 16 / 19 / 20 / 21 / 26 / 32-39 轮）：说明 AI 在"知识广度"和"工程执行"上确实给了好建议
- **有的轮次我大幅纠偏**（第 2 / 6 / 13 / 25 轮）：说明 AI 在"产品判断"上有结构性盲区——倾向技术维度、友好场景、安抚式话术

这种**有采纳有拒绝**的协作模式，才是真实的人机协作——而不是"假装 AI 总是错的"或"假装 AI 总是对的"。

**第 32-39 轮（实机调优阶段）的特别观察**：进入"修产品"模式后，PM 输入的 prompt 都极简短（一两句话提需求），AI 主导方案设计 + 实施。这个阶段的协作模式更像"产品经理 + 工程师"而不是"业务方 + AI"——说明 AI 在"工程化执行"上已经能承接 PM 的全部需求翻译，PM 的价值集中在"定义问题"和"判断结果"两端。

PM 的价值在每个决策的开头（定义问题、给约束、立指标）和每个决策的末尾（用户感知校验、Reversible 判断、Who-wins-loses 评估）。AI 的价值在中间（穷举方案、知识广度、执行效率）。
