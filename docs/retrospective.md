# 构建复盘

## 需求拆解思路

题目核心不是做一个“旅游 App”，而是做一个“座舱内 AI 语音交互原型”。因此我把需求拆成两层：

- 产品层：把功能一收敛成 6 个关键帧，并确保每一帧在大屏上都有清晰反馈。
- 实现层：用一个可控状态机驱动流程，而不是依赖不可预测的自由生成对话。

主要取舍：

- 保留：需求发起、信息确认、方案生成、方案调整、方案确认、保存分享的完整链路。
- 弱化：真实语音识别、地图能力、酒店机票搜索、复杂分支树。
- 放弃：数据库、账号体系、真实分享链路。

这样做的原因是优先确保题目要求的演示完整性，而不是在非关键能力上增加复杂度。

## 技术选型理由

- 前端选择原生 `HTML/CSS/JavaScript`
  原因是启动成本低、依赖少、适合快速做一个可运行的交互原型。
- 后端选择 `FastAPI`
  原因是接口定义清晰，适合用 Python 快速搭建状态机和规则服务。
- 数据层使用内存态
  原因是本题是 Demo，不需要持久化，避免过度设计。

备选方案考虑过：

- React：更适合复杂组件，但当前原型规模不大，原生前端足够。
- Flask：也能完成，但 `FastAPI` 的类型约束和接口结构更利于后续扩展。

## AI 协作过程

关键协作方式不是“让 AI 一次性生成全部代码”，而是把它拆成几个明确任务：

- 先根据题目抽取功能一和交付要求。
- 再把实现约束固定为 `FastAPI + 静态前端 + Python 状态机`。
- 再要求 AI 输出清晰的数据模型、接口设计和状态流转规则。
- 最后用测试校验主流程是否真的闭环。

有效的指令特点：

- 明确限制实现边界，例如“不接真实旅游 API，只做 Mock 逻辑”。
- 明确用户场景，例如“远距离大屏、语音为主、低信息密度”。
- 明确结构要求，例如“必须有 6 个关键帧，必须有会话状态机”。

## 问题与解决

### 1. 题目要求偏产品，但仍需写可运行代码

解决方式：

- 用状态机承接产品流程，用前端原型承接展示表达。
- 把“产品完整性”放在“技术复杂度”前面。

### 2. 没有真实旅游数据源

解决方式：

- 使用规则模板生成 Mock 行程。
- 让调整逻辑聚焦在预算、节奏、城市权重、偏好几个关键变量上。

### 3. 座舱场景容易做成普通聊天页

解决方式：

- 强制前端把“AI 状态、关键信息、每日行程、语音回放”拆成不同展示区域。
- 控制信息密度，突出关键帧和下一步动作。

## 反思与改进

如果继续迭代，我会补以下能力：

- 增加真实 LLM 接口，把规则与自然语言生成结合起来。
- 接入真实 POI / 路线 / 天气接口，让方案更可信。
- 增加“多人出行”“亲子”“高铁/飞机偏好”等更多约束。
- 把分享链路做成真实二维码或 H5 卡片。

对 AI Coding 工作流的新理解：

- AI 很适合快速搭骨架、列接口、补样式和写样板代码。
- 真正决定结果质量的，仍然是需求收敛、边界约束和验收标准。
- 如果不先把状态机、数据模型和演示路径定清楚，AI 很容易生成一个看起来完整但难以演示的方案。
